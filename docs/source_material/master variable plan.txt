  Short Summary
  The implemented model is still mostly flat and workable, but the input boundary is uneven: YAML has more source inputs
  than the GUI exposes, some GUI/config-manager paths omit fields, and several fields blur user-owned inputs with
  calculated/output values. The recent _rate rename group fits the style guide and gives a clean foundation.

  The biggest current risks are not the flat Inputs dataclass itself. They are ambiguous Year 1 semantics, ACA subsidy
  input naming/meaning, and fields that appear in YAML/tests but are missing from the GUI save/load surface.

  5. Flat Vs Grouped Containers

  Flat Inputs is acceptable for now. The field count is high, but the current loader, tests, GUI, and engine all expect
  one simple object, and a grouping refactor would touch too many boundaries before the input meanings are stable.

  Grouped containers would solve discoverability, category validation, and GUI/YAML alignment. They would also make it
  harder to confuse ACA inputs with tax outputs or Year 1 seeds with projection-wide assumptions.

  Grouped containers would create migration risk: loader changes, GUI mapping changes, fixture churn, and likely
  compatibility decisions. The current evidence supports future groups for personal, balances, spending, year1, rates,
  tax, aca, social_security, rmd, and draw_strategy, but not yet.

  Safest trigger for grouping: after rename-only cleanup and after ACA/Year 1 source inputs are defined, but before
  adding a larger second wave of feature inputs.
